# Contributing to Lisible Documentation

Thank you for improving the official Lisible documentation. Every contribution should make the documentation more accurate, easier to navigate, and equally useful in French and English.

## Edit a page on GitHub

Each documentation page exposes **Edit this page** / **Modifier cette page**. The link opens the exact MDX source in GitHub's editor:

```text
https://github.com/didntchooseaname/lisible-docs/edit/main/src/content/docs/<locale>/<page>.mdx
```

When signed in:

1. Edit the source and preview the diff.
2. Choose **Propose changes**.
3. GitHub creates a branch or fork when required.
4. Open a pull request with a concise explanation and screenshots when rendering changes.

The public repository must exist on the `main` branch before edit URLs can resolve. Never commit generated `dist/`, `node_modules/`, `.astro/`, `.output/`, caches, secrets, or local environment files.

## Develop locally

Requirements: Git and [Bun](https://bun.sh).

```bash
git clone https://github.com/didntchooseaname/lisible-docs.git
cd lisible-docs
bun install
bun run dev
```

The development server provides a safe Pagefind placeholder. The real full-text index is generated by the production build.

## Documentation structure

Pages live in mirrored trees:

```text
src/content/docs/fr/<category>/<page>.mdx
src/content/docs/en/<category>/<page>.mdx
```

The sidebar is generated from frontmatter. Keep matching relative paths, category, order, badge intent, headings, internal references, code samples, footnotes, diagrams, and feature coverage in both locales. Translate prose and labels naturally.

Use `@/*` for regular imports from `src/*` and `#src/*` in modules loaded directly by Astro configuration. Do not navigate the source tree with parent-relative `../` imports.

Supported authoring tools include Markdown, MDX components, tabs, steps, spoilers, callouts, tables, task lists, footnotes, KaTeX, Mermaid, draw.io, image captions, code frames, line numbers, and internal links. Prefer an internal documentation link when the concept is explained locally; use an authoritative external reference for framework behavior.

## Navigation and i18n

- Keep every internal link absolute from the site root and point it to a published route.
- Verify sidebar and previous/next links in French and English.
- Language changes must use Astro client navigation without a full refresh.
- Keep documentation URLs authoritative. `/fr/` and `/en/` are the stable landing URLs; the `x-default` `/` entry defaults to English and may localize itself to French in place. The 404 uses the same stored/browser preference without navigating.
- Keep page titles, descriptions, navigation labels, search, 404 content, and accessibility text localized.

## Quality checks

Run both checks before opening a pull request:

```bash
bun run check:all
bun run preview
```

`check:content` enforces FR/EN parity, MDX usage, and internal links. The build validates every static route and generates Pagefind, sitemap, robots, and text exports. In the browser, verify the changed page, its mirror, the sidebar after an Astro transition, keyboard interaction, light and dark themes, responsive navigation, code blocks, and a 404 route. Normal pages must leave both terminal and client consoles free of warnings and errors.

## Pull request checklist

- Explain what was incorrect or missing and what the change improves.
- Update both locale files or explain why a change is language-specific.
- Include screenshots for layout, component, diagram, or theme changes.
- Confirm `bun run check:all` passes.
- Keep the pull request focused and do not include generated output.
